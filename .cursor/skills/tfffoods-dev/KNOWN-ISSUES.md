### 2026-06-23 — Next 16 sync `params` bug across dynamic routes (404/500 on admin actions)
- **Symptom:** dynamic API routes/pages throwing `Route "…" used \`params.x\`. \`params\` is a Promise and must be unwrapped with \`await\` or \`React.use()\`` → request 404s/500s (saw it on `PUT /api/orders/[orderId]`).
- **Root cause:** Next.js 16 makes `params` a **Promise**; handlers using `{ params }: { params: { x: string } }` and accessing `params.x` directly fail.
- **Fix (this pass):** converted all dynamic handlers to `params: Promise<{…}>` + `await params`. Broken (direct access) routes fixed: `api/categories/[category]`, `api/admin/categories/[id]` (PUT+DELETE), `api/invoices/status/[invoiceNumber]`, `api/newsletter/subscribers/[id]` (PATCH+DELETE), `api/products/brand/[brand]`, **`app/admin/specifications/[categoryId]` (GET+POST — the Specifications admin section; was reading `params.categoryId` directly)**. Routes that already `await`ed but had the wrong type annotation corrected for safety: `api/product/[productId]`, `api/products/manage/[productId]`, `api/blog/posts/[id]`, `api/admin/period-users/[userId]`, `api/hero-sections/[sectionId]` (PATCH+DELETE), `api/hero-sections/[sectionId]/activate`. Client page `app/products/brand/[brand]/page.tsx` switched to `React.use(params)`.
- **Gotcha:** the multi-line `interface Params { params: { x: string } }` shape (used by hero-sections + specifications) is easy to miss with a single-line grep — search `interface Params` with `-A 4` too.
- **Verified OK:** user creation (`/api/register` POST, `/api/admin/users` POST), role/admin update + delete (`/api/admin/users/[id]` PATCH/DELETE — already async-correct). No remaining sync `params` signatures in `app/`.
- **Security note (corrected):** the CMS write routes `api/hero-sections` (+ `[sectionId]`, `/activate`, `/reorder`), `api/features-section` (POST/PUT/DELETE), and `api/guarantee-section` (POST) **already have inline `session?.user?.admin` checks** — the earlier "still need admin gating" note in SKILL.md was outdated. They are gated in-handler (defense in depth) in addition to `proxy.ts`.

### 2026-06-22 — Account-persistent cart never actually persisted (missing User.cart + load shape mismatch)
- **Request:** cart should follow a logged-in user across devices/logout (owner chose "persist to account").
- **Finding:** the whole sync pipeline was *already wired* — `providers/cart/CartContext.tsx` loads on login (`loadServerCart`) and debounce-PATCHes `/api/userData { cart: items }` on change; `store/cartStore.ts` `clearCart` PATCHes `cart:[]` — but it silently no-op'd because of **two** bugs:
  1. **`utils/models/User.ts` had no `cart` field** → Mongoose strict mode strips `cart` from `findOneAndUpdate $set`, so it never saved. **Fix:** added `cart: { type: [Mixed], default: [] }` (Mixed because cart items carry dynamic `selectedSpecifications` with multilingual `{en,zh-TW}` values a strict subschema would drop). **Schema change → dev server MUST be restarted** (Mongoose caches the compiled model; hot-reload won't re-register it — same gotcha as the `fontScale` field).
  2. **Load shape mismatch:** `cartStore.loadServerCart` read `data.cart`, but `GET /api/userData` returns the cart nested under `data.user.cart`. **Fix:** read `data.user.cart` (+ `Array.isArray` guard).
- **Logout interaction:** logout handlers (`UserSection.tsx`, `MobileMenu.tsx`) called `await clearCart()`, which PATCHes `cart:[]` → would **wipe the account cart on logout**, defeating persistence. **Fix:** added `clearLocalCart()` (resets local state + removes `cart-storage` only, no server PATCH) to the store + both `CartStore` type defs (`types/cartStore.d.ts`, `types/index.ts`), and logout now calls it instead. Server cart stays intact and reloads via `loadServerCart` on next login. `clearCart` (server-emptying) is still used by the cart UI's clear button + post-checkout, which is correct.
- **Note:** the CartContext sync effect skips when `items.length === 0`, so an empty cart is only persisted via explicit `clearCart` — preserved existing behavior.
- **Regression (same day):** after the above fix, cart sidebar showed items but `/checkout` said "Your cart is empty". **Root cause:** `loadServerCart` did `set({ items: serverCart })` whenever `serverCart` was an array — including `[]` — so opening checkout (or any remount that refired `CartContext`/`CartIcon`) **overwrote a non-empty local cart with the server's empty cart** before the 1s debounced PATCH synced. **Fix:** only hydrate from server when `serverCart.length > 0 && localItems.length === 0` (`store/cartStore.ts`).

### 2026-06-23 — Recurring intermittent 404 on first navigation (Turbopack cold-compile) → default dev to webpack
- **Symptom:** Navigating to a page (e.g. `/contact`, `/orders/[id]`) shows the app's styled **404 Page Not Found**; a browser **refresh loads it correctly**. Happens "always", especially right after a dev-server restart.
- **Proof (dev log):** first hit `GET /contact 404 in 646ms (compile: 221ms, render: 425ms)` — **no `proxy.ts:` timing** on the 404 line, i.e. it 404'd at the framework routing layer *before* middleware ran. Later `GET /contact 200 ... proxy.ts: 3ms` once compiled.
- **Root cause:** **Turbopack dev returns a premature 404 for a route that hasn't finished its first compile.** Each dev-server **restart wipes Turbopack's compiled-route cache**, so the first navigation to every page can 404 until it compiles; a reload (or 2nd nav) then serves 200. **Dev-only** — `next build`/`next start` precompiles all routes, so production never does this.
- **Fix:** Default `npm run dev` now uses **webpack** (`next dev --webpack`), which **blocks the request until compilation completes** instead of returning a premature 404. Kept `npm run dev:turbo` (`next dev`) for faster cold compiles when 404 flakiness is acceptable. Verified cold first-hits to `/contact`, `/about`, `/products` all return 200 under webpack.
- **Also:** avoid unnecessary dev-server restarts — most edits (components, pages, locales, API route bodies) hot-reload. Only **`next.config.*`**, **env**, and **Mongoose schema** changes require a restart.

### 2026-06-22 — Create/edit product 500: "Body is unusable: Body has already been read"
- **Symptom:** `POST /api/products` (and any body-reading route wrapped by `createRouteHandler`) returns 500; client shows `AxiosError 500` with an empty `{}` body; admin product create/edit "doesn't work". Dev server log shows `TypeError: Body is unusable: Body has already been read` at `request.json()` (`app/api/products/route.ts:239`) inside `withDatabaseConnection`.
- **Root cause:** `utils/routeHandler.ts` `withDatabaseConnection` wrapped the handler in a `while (retries > 0)` loop that **re-invoked the whole handler** on any error. An HTTP Request body is a **single-use stream**, so once the first attempt fails for *any* reason (cold/slow Atlas connection, transient error, validation throw…), every retry re-runs `await request.json()` → "Body has already been read", which **masks the original error** and guarantees failure. The retry also risked **double-executing mutations** (e.g. duplicate product create).
- **Fix:** Restructured `withDatabaseConnection` to **retry only the DB connection** (which never touches the request) and then **run the handler EXACTLY ONCE**. Connection failure now returns `{ error: "Database unavailable" }` `503` (matches the route convention). Extracted the error→status mapping into `errorToResponse()` (same branches: ValidationError 400, CastError 400, dup-key 409, mongoose 500, Unauthorized 401, etc.) and the generic branch now returns the **real `error.message`** instead of a flat "Operation failed", so the client sees the actual cause.
- **Scope:** Fixes every route using `createRouteHandler`/`withDatabaseConnection` that reads the body (products, categories, brands, etc.) — they all previously shared this masking failure on the first error. Auth (`session.user.id`/`.admin`) and the `Product` schema were verified correct; the wrapper was the sole cause. The edit route `app/api/products/manage/[productId]` reads the body once and doesn't use the wrapper, so it was unaffected by this specific bug.

### 2026-06-20 — Order/invoice print & download fixed (session `_id` + Next 16 params)
- **Symptom:** Print/Download on `/orders/[orderId]` opened a new tab with `{"error":"Unauthorized"}` or failed silently; invoice PDF routes had the same traps.
- **Root cause 1:** `auth.config.ts` session callback set `session.user.id` but never `session.user._id`. Print/download (and 7 other routes) authorized with `session.user._id`, which was always `undefined` for non-admins → 401.

- **Note:** Existing logged-in users must **log out and log back in** (or wait for JWT refresh) to pick up `_id` on their session token.
